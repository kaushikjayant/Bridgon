RTL file list

The RTL file list (.f) references all SystemVerilog (IEEE 1800) and VHDL (IEEE 1076) source files that comprise the SoC or subsystem design. This is the primary input to the Verific parser and elaborator.

The file list must include the complete design hierarchy — from the chip top-level module down through all subsystems and IP instances — so that Verific can perform full static elaboration and resolve all parameter overrides at every level.

SystemVerilogVHDLIEEE 1800IEEE 1076
Top spec memory map (XLSX)

The memory map XLSX is the single authoritative source for what is in scope for the current SoC or subsystem spec. Each row describes one IP instance with: IP name, spec repository path (git URL), .rdb register file link, start address, end address, slot size, and address space identifier.

The pipeline reads this file to populate spec directories from repository paths and to establish the address layout for memory map generation and .vmap register instance construction.

openpyxladdress layoutspec repo paths
Non-RTL parameter database

A hierarchical JSON file (version-controlled alongside documentation source) recording associations between documentation-specific parameters — IP revision identifiers, feature flags, chip-tier labels, protocol variant names — and specific IP, subsystem, and SoC instances.

Values are keyed by chip prefix (e.g. IMX95) and instance name, enabling the merge step to extract the correct non-RTL value for each instance in the target spec. This replaces the prior unstructured spreadsheet approach with a schema-governed, diffable data source.

Verific RTL analysis & elaboration

Uses Verific Design Automation's SystemVerilog/VHDL parser and RTL elaborator — an industry-grade tool supporting the complete IEEE 1800 and IEEE 1076 standards. Analysis creates parse trees and performs type inferencing; RTL elaboration then binds instances to modules, resolves library references, propagates defparam assignments, unrolls generate statements, and resolves all hierarchical names.

After elaboration, every parameter override at every level of the design hierarchy is fully resolved. A Python script using Verific's API walks the elaborated hierarchy and dumps module_parameters.json — capturing resolved parameter values, instance paths, module names, and register definitions for every IP instance in the design. Replaces the previous inaccurate LEC-based extraction approach.

VerificRTL elaborationdefparam resolutionmodule_parameters.jsonPython API
Spec repo population

In parallel with Verific analysis, the pipeline reads the memory map XLSX and uses git clone / git pull to populate each IP spec directory from its repository path. This ensures the complete spec_source tree — maps/, topics/, inst/, params/, regs/, images/ — is present locally before assembly begins.

02Parameter merge
RTL + non-RTL parameter merge → .vmap
Combines Verific RTL params + non-RTL DB (chip-prefix keyed) into a unified value map with register instance elements per IP
RTL + non-RTL merge → .vmap

The merge step combines module_parameters.json (RTL-sourced, from Verific elaboration) with the non-RTL parameter DB (doc-specific params, keyed by chip prefix) to produce a unified .vmap (value map) file — the central artifact of the pipeline.

For each IP instance in the memory map, the pipeline creates a register instance element in the .vmap. Each register instance links its .rdb register database file and carries both RTL and non-RTL parameter bindings as <param-binding> elements. Multiple register instances can reference the same .rdb with different bindings — enabling per-instance parameter variation of shared register sets.

Top DITA map generation

Auto-generated from the .vmap and memory map XLSX. Iterates over register instances to insert <topicref format="ditamap"> elements referencing each IP spec's ip.ditamap in the correct hierarchical order. AI-authored SoC chapter topics are inserted at the front of the map.

.ditamaptopicrefhierarchical assembly
Top publishing manifest (.pub)

The .pub publishing manifest defines instance selectors (referencing a .vmap file or a named instance) and hierarchical resolve groups that govern how register information is merged and deduplicated during PDF and WebHelp publishing.

When the same IP appears at multiple base addresses, the resolve group ensures common register content appears only once, with per-instance address information consolidated in a generated memory map table. The top-level .pub references IP-level .pub files via <include-proc>, building a complete hierarchical publication tree.

.pubresolve groupsinstance selectorsregister deduplication
Top parameter definitions (.prm)

Consolidated SoC-level .prm declaring all SoC-specific parameters — type (boolean / integer / float / string / enumerated), default value, allowed range, and RTL vs non-RTL source. IP-level .prm files remain in their spec directories and are not duplicated at the SoC level, maintaining a clean single-source inheritance model.

.prmboolean/integer/float/string/enumRTL vs non-RTL
Chip-specific DITA sections

For each IP instance in the SoC, the pipeline auto-generates chip-specific DITA topic files — a chip intro topic and a chip configuration topic — and inserts references to these into the top-level DITA map and the .vmap. Enables SoC-specific narrative and configuration content without modifying the reusable IP spec source.

Register spec validation

Verific elaboration extracts complete register definitions alongside parameter values — register names, bit-field names, bit positions, reset values, access types. These are cross-validated against the corresponding .rdb files in each IP spec. Discrepancies (missing registers, reset value mismatches, bit-range errors, access type differences) are surfaced as structured reports in the VSCode plugin, catching a class of errata before tape-out.

AI SoC chapter authoring

A Claude API agent (claude-sonnet) receives: (1) the relevant section of the SoC architecture specification document, (2) a DITA structural template defining required topic type and section hierarchy, and (3) the merged .vmap parameter context providing accurate instance names, base addresses, and feature flags.

The agent outputs DITA-conformant XML: <concept> topics for architectural narratives (clocking overview, power domains, boot sequence) and <reference> topics for tabular data (interrupt matrix, memory map summary). Parameter context ensures cross-references to instance names and addresses are accurate.

DITA-OT wrapper + param resolver

A Python wrapper around DITA-OT 3.x runs a resolution preprocessing pass before standard transformation. All DITA topics, maps, and register files support a custom applies-when attribute that can conditionalize any element on any instantiated parameter value (RTL or non-RTL).

The resolver evaluates these conditions against the active .vmap and produces fully resolved, parameter-free DITA output — filtering inapplicable content and substituting parameter values inline. The resolved DITA is then passed to standard DITA-OT for PDF2 or WebHelp-Responsive transformation.

DITA-OT 3.xapplies-whenparam resolutionPDF2WebHelp-Responsive
.rdb → DITA transform

Each .rdb register database file is transformed into a DITA <reference> topic. The transform auto-generates: (1) an SVG bit-field diagram showing register layout with field names and bit positions, and (2) a structured table with field name, offset, reset value, access type, and description columns. Output is embedded directly in the resolved DITA.

Parameterized fields and conditional register blocks in the .rdb (gated by applies-when attributes) are resolved against the instance's .vmap bindings before transformation.

03   Custom file types

Proprietary File Type System

Four file types extend the standard DITA authoring model to support parameter-driven publishing across the three-tier spec hierarchy.

.prm
Parameter definition file
Declares all parameters used in a spec tier — data type, default value, allowed range or accepted value set, and RTL vs non-RTL source. A global .prm covers shared publishing parameters that need not be redeclared per spec. IP .prm files are not duplicated at subsystem or SoC level.
boolean integer float string enumerated RTL / non-RTL
.vmap
Value map — instance bindings
Binds resolved parameter values to the parameters declared in the .prm. Contains register instance elements, each linking a .rdb file and carrying per-instance parameter bindings. Multiple register instances can reference the same .rdb with different bindings — enabling per-instance variation of shared register sets.
param-override reg-instance param-binding base-address
.pub
Publishing manifest
Defines instance selectors and hierarchical resolve groups governing how register information is merged and deduplicated across IP instances during publishing. Subsystem and SoC .pub files reference IP .pub files via include-proc elements, building the full publication tree.
resolve-group instance-selector include-proc address-instance
.rdb
Register database
Proprietary XML analogous to IPXACT. Stores complete register definitions: name, offset, reset value, access type, and full bit-field specs. Supports parameterized fields and conditional register blocks via applies-when attributes. Transformed to DITA reference topics with SVG bit-field diagrams during publishing.
register field applies-when param-condition IPXACT-like

Python verific_extractor.py 
import verific_db as vdb
import json

def extract_module_parameters(top_module: str, filelist: str, work_dir: str) -> dict:
    # Compile and elaborate the RTL hierarchy
    vdb.analyze_files(filelist=filelist, work_dir=work_dir)
    vdb.elaborate(top=top_module)

    result = {}
    design = vdb.get_design(top_module)

    for inst in design.iter_instances(recursive=True):
        module = inst.get_module()
        params = {}
        for param in module.get_parameters():
            # Post-elaboration: fully resolved override value
            params[param.name] = {
                'value':       inst.get_param_value(param.name),
                'type':        param.data_type,
                'default':     param.default_value,
            }
        result[inst.full_path] = {
            'module_name':  module.name,
            'parameters':   params,
            'registers':    extract_register_defs(inst),
        }
    return result

06   Generative AI

AI-Powered Chapter Authoring

Claude API agent extracts architecture content from SoC specs and produces DITA-conformant chapter topics, parameterised with accurate instance data from the .vmap.

Python soc_chapter_agent.py 
import anthropic, json

client = anthropic.Anthropic()

SYSTEM = """You are a semiconductor documentation engineer at Bridgon.
Generate DITA 1.3 conformant XML chapter topics for SoC Reference Manuals.
- Use <concept> for architectural overviews and narratives
- Use <reference> for tabular data (registers, addresses, interrupts)
- Use only values present in the architecture spec and .vmap context
- Never invent register values, addresses, or instance counts
- Output only valid DITA XML — no prose wrapper, no markdown"""

def generate_chapter(arch_spec_section: str, chapter_type: str,
                      vmap_context: dict, template: str) -> str:
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=4096,
        system=SYSTEM,
        messages=[{"role": "user", "content": f"""
Chapter type: {chapter_type}
Template: {template}
Parameter context: {json.dumps(vmap_context)[:3000]}
Architecture spec: {arch_spec_section[:6000]}
Generate the complete DITA topic."""}]
    )
    return response.content[0].text
